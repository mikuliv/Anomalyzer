# Anomalyzer / Филин

Репозиторий объединяет историческую версию проекта Anomalyzer и новую исследовательскую платформу «Филин».

## Назначение проекта

Цель проекта - переход от простого ML-классификатора сетевых аномалий к платформе мониторинга, анализа и поддержки расследования инцидентов информационной безопасности.

Новая часть проекта должна связывать лабораторный сбор событий, нормализацию, построение признаков, обучение моделей, backend-детекцию, карточки инцидентов, сопоставление с MITRE ATT&CK и подготовку Sigma-кандидатов.

## История проекта

`Anomalyzer-main/` - исходная версия проекта, выполненная как выпускная квалификационная работа СПО по специальности 10.02.05 «Обеспечение информационной безопасности автоматизированных систем».

Эта папка сохраняется как историческая база: она показывает исходную идею, начальную ML-модель и демонстрационный контур колледжной ВКР. Новая разработка ведется отдельно в `filin/`.

## Структура репозитория

- `Anomalyzer-main/` - историческая версия проекта Anomalyzer.
- `filin/` - новая исследовательская платформа.
- `filin/backend/` - FastAPI backend, прототип детекции и обработки инцидентов.
- `filin/lab/` - лабораторный стенд, сценарии, manifest-разметка и инструменты подготовки датасета.
- `filin/ml/` - построение признаков, заготовки обучения, оценки и экспорта моделей.
- `filin/collectors/` - заготовки коллекторов Zeek, Suricata и CSV.
- `filin/datasets/` - описание датасетов, схем и маленькие учебные примеры.
- `filin/docs/` - архитектура, моделирование, дорожная карта и статус компонентов.

## Что такое «Филин»

«Филин» - развитие идеи Anomalyzer в сторону платформы. Вместо отдельного классификатора проект постепенно формирует полный контур:

- сбор и нормализация сетевых событий;
- построение window-level и flow-level признаков;
- обучение и оценка моделей обнаружения;
- API детекции и обработки инцидентов;
- сопоставление с MITRE ATT&CK;
- генерация Sigma-кандидатов;
- проверка сценариев и правил на лабораторном стенде.

## Текущий pipeline

```text
лабораторный стенд
    ↓
scenario_manifest.yaml
    ↓
execution_events.jsonl
    ↓
traffic_events.jsonl
    ↓
normalized_events.jsonl
    ↓
windows_v0_1.csv / flows_v0_1.csv
    ↓
обучение и оценка моделей
    ↓
карточка инцидента
    ↓
MITRE ATT&CK
    ↓
Sigma-кандидат
```

Manifest описывает план и разметку сценариев. `execution_events.jsonl` хранит служебные события выполнения. `traffic_events.jsonl` хранит учебные события активности внутри сценариев. `normalized_events.jsonl` приводит события к единому формату. `windows_v0_1.csv` и `flows_v0_1.csv` являются агрегированными датасетами признаков для ML.

## Текущее состояние реализации

- Backend-прототип на FastAPI готов для базовых endpoint-ов.
- Есть прототип обработки инцидентов.
- Есть первичное сопоставление классов с MITRE ATT&CK.
- Есть генерация Sigma-кандидатов.
- Есть Docker-лабораторный стенд.
- Есть manifest-разметка сценариев и natural-режим расписания.
- Есть mock/execute-контур выполнения сценариев.
- Есть генерация `execution_events.jsonl`, `traffic_events.jsonl`, `normalized_events.jsonl` и `dataset_report.md`.
- Есть модуль feature extraction: feature catalog, schema, validators, window-level и flow-level CSV.
- Есть учебные example CSV и CLI-проверка их структуры и базовой арифметики.

В последнем проверочном mock-прогоне формировалось 13 сценариев, 454 traffic events и 480 normalized events.

## Ограничения

Проект не предназначен для атак на внешние системы. Все attack-сценарии являются безопасными лабораторными имитациями и должны выполняться только в изолированной среде.

Реальные боевые логи, чувствительные данные, большие датасеты, PCAP и артефакты моделей в репозиторий не добавляются. `filin/lab/output/`, `*.jsonl`, `*.csv`, `*.pcap`, `*.pcapng`, модели и большие артефакты обучения не хранятся в Git. Исключение сделано только для маленьких учебных примеров в `filin/datasets/examples/`.

Mock-режим нужен для проверки pipeline. Он не заменяет реальный сбор трафика в Docker/VMware-стенде и не должен использоваться как финальный датасет для оценки моделей.

## Дальнейшее развитие

- Расширить сбор событий через Zeek и Suricata.
- Уточнить схему признаков на реальных flow и HTTP/DNS логах.
- Провести train/test split до балансировки классов и обучения scaler/encoder.
- Сравнить MLP, RandomForest, XGBoost/LightGBM и AutoEncoder.
- Оценить precision, recall, F1, ROC-AUC/PR-AUC, confusion matrix и ошибки по классам.
- Подключить постоянное хранилище, dashboard или SIEM export.
- Подготовить демонстрационный сценарий: событие -> детекция -> инцидент -> MITRE -> Sigma -> проверка.

## Филин v0.2: реальный Docker traffic collection

`dry-run` проверяет расписание без сетевой активности. `mock` создаёт синтетические события для проверки pipeline. `docker` выполняет безопасные HTTP, DNS-like и TCP-действия только между разрешёнными контейнерами изолированного стенда. Эти события наблюдаются со стороны `traffic-client`; они не являются независимыми сетевыми flow. Полноценные conn/http/dns flow появятся после подключения Zeek/Suricata.

## Филин v0.2.1 — Docker-to-Docker evaluation

Этап сравнивает независимые Docker-прогоны, контролирует происхождение датасетов по run ID и SHA-256, оценивает baseline-модели и анализирует смещение признаков. Результаты относятся только к воспроизводимости лабораторного стенда.
